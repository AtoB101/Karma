## 三、EIP-712 签名安全 — Karma 核心依赖

### EIP-712 常见漏洞

1. **Domain Separator 配置错误**
   - chainId 不匹配 → 跨链重放
   - verifyingContract 写错 → 跨合约重放
   - 必须严格校验 domain separator

2. **签名可塑性（Signature Malleability）**
   - 使用 OpenZeppelin ECDSA 库的 `tryRecover` 而非裸 `ecrecover`
   - 必须检查 `s` 值范围（EIP-2）

3. **重放攻击**
   - 必须有 nonce 机制
   - 考虑 deadline/timestamp 限制签名有效期
   - ERC-7803 引入了 signing domains 在跨账户共享私钥时防重放

4. **签名类型混淆**
   - EIP-712 的 `typeHash` 必须严格匹配
   - 不同类型的数据结构可能导致签名被滥用

5. **前端钓鱼风险**
   - 用户看到的是哈希，不是明文
   - 钱包必须正确解析和显示 EIP-712 类型数据

### Karma 应该做什么
- ✅ AuthTokenManager 用 EIP-712（已有）
- ⚠️ 必须审查 domain separator 是否包含 chainId + verifyingContract
- ⚠️ 必须加入 nonce 防重放
- ⚠️ 必须检查签名有效期（deadline）
- ⚠️ 用 OpenZeppelin ECDSA 库，不用裸 `ecrecover`

## 四、Access Control 深度分析（Karma 最高优先级）

### 为什么 Access Control 是 #1 杀手？
2024 年 $953M 损失全来自访问控制缺陷，主要攻击路径：

1. **初始化函数可被任何人调用**（88mph 事件）
2. **onlyOwner 缺失或配置错误**
3. **函数可见性设为 public 而非 restricted**
4. **代理合约的初始化漏洞**
5. **私钥泄露 → Admin 权限被盗**

### Karma 必须做到
- [ ] 所有敏感函数必须有明确的 access control modifier
- [ ] 使用 OpenZeppelin AccessControl 而非手写 RBAC
- [ ] Admin 权限多签或 timelock
- [ ] CircuitBreaker 的 pause/unpause 权限严格控制
- [ ] 初始化函数必须带 `initializer` modifier（如果用 proxy）
- [ ] 部署后立即 renounce 不必要的 admin 权限
