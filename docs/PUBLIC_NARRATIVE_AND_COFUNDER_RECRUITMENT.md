# KARMA — 公开叙事、场景落地与核心联创招募（公开版）

> 本文档面向：**潜在用户、开发者、投资人、核心联创候选人**。  
> 技术细节以仓库内合约与文档为准；**不构成投资建议或法律意见**。

---

## 一、我们是谁（一句话 + 一层愿景）

**一句话（主叙事）**  
KARMA 面向 **高频小额、可数字化交付** 的协作——**API 按次调用、技术外包、跨境小额数字服务、数据标注批次**——提供 **非托管锁款、账单与证据对齐、可审计结算路径**；执行由 Agent / 人完成，**钱与规则由链上状态机与公开契约守住**。

**愿景（向上的一层，不改变当前切口）**  
当 AI 与 API 让交易更碎、更自动化，**信任不应只依赖人情或平台黑箱**。KARMA 从 **最小、最高频、最易标准化** 的数字协作场景做起，逐步成为 **「数字交付 + 结算」的可审计基础设施**——开源优先，商业可并行。

---

## 二、我们解决什么问题（痛点）

| 角色 | 典型痛点 |
|------|----------|
| **卖方（开发者、标注团队、API 卖家）** | 怕白干：交付了对方不付、反复改需求、扯皮质量。 |
| **买方（调用方、发包方）** | 怕被骗：先付款对方跑路、交付物对不上、跨境难追责。 |
| **双方** | 小额高频下，传统合同/大平台 **太重**；纯聊天成交 **无保障**；仅「转账」**无法绑定交付与争议材料**。 |

**KARMA 的命题**  
把 **「钱何时动」** 与 **「交付与证据是否对齐」** 绑在同一套 **可公开验证的规则与状态** 上；**主观质量与反作弊** 不放在公开仓黑箱，而由集成方与（可选）私有服务在边界外处理。

## 三、首发场景（公开定义）

以下四类 **并列** 为第一阶段主战场；对外沟通可四选一突出，但**技术底座同一套**。

### 1. API 按次 / 计量调用

- **特征**：高频、小额、可日志化。  
- **KARMA 价值**：锁款 / 账单与 **调用结果摘要或哈希** 对齐，结算与争议有据。

### 2. 技术外包（远程数字交付）

- **特征**：建站、自动化、小工具、远程实施。  
- **KARMA 价值**：确认前锁款；交付物与 **证据包 / proof 语义** 对齐；争议路径可追溯。

### 3. 跨境小额数字服务

- **特征**：陌生人、时差、收款摩擦。  
- **KARMA 价值**：链上规则可审计；**具体法域合规与持牌路径**由集成方与合作伙伴处理，KARMA 聚焦 **数字交付 + 结算绑定**。

### 4. 数据标注市场（批次 / 众包）

- **特征**：按条/框/批次计费，易质量纠纷。  
- **KARMA 价值**：**批次级账单 + 证据指针**；**原始敏感数据不进链、不进公开仓**（哈希、批次 ID、存储 URI 与访问控制由集成方负责）。质量与作弊策略 **不** 以黑箱形式写入公开仓库。

## 四、KARMA 做什么 / 不做什么（边界）

| 我们做 | 我们不做（第一阶段公开承诺） |
|--------|------------------------------|
| 非托管锁仓、账单生命周期、结算与争议链上路径（以 `NonCustodialAgentPayment` 等为真值） | 不做「全自动 AI 法院」式承诺 |
| 执行收据、证据包、与现有 `proofHash` 语义对齐的适配层（`trusted_agent_runtime`） | 不在公开仓发布私有评分权重、反作弊细则 |
| 静态门户、Console 壳、开发者文档、OpenAPI 契约与测试网脚本 | 不替代集成方完成全部合规与客户服务 |
| 开源（AGPL-3.0-only）+ 文档中的商业许可路径说明 | 不承诺无风险、不承诺保本 |

## 五、落地现状（仓库里「已经有什么」）

便于联创与贡献者 **对齐预期**：下列为截至本仓库主线能力的 **公开事实**（随版本迭代以 README 与 CHANGELOG 为准）。

| 层级 | 状态 | 说明 |
|------|------|------|
| **链上结算内核** | 已有 | `contracts/`：`NonCustodialAgentPayment`、`SettlementEngine`、配套模块与 Foundry 测试。 |
| **链下可信适配** | 已有 | `trusted_agent_runtime`：收据、证据包、结构校验、结算计划映射、`trace_id`、运营级 pause/冻结与幂等键思路；测试网脚本与运行手册。 |
| **静态产品入口** | 部分已有 | `apps/website`、`apps/console` 壳、`apps/developer-portal`；`apps/agent-service-guard`（门户 + WalletConnect 登录 + Studio 壳）。 |
| **公开 HTTP 产品化** | 契约与脚手架为主 | `openapi/karma-public-console-api.yaml`、`openapi/karma-v1.yaml`；**完整生产级 BFF、索引、多租户** 需联创阶段补齐。 |
| **SDK / CLI** | 脚手架 | `packages/sdk`、`packages/cli`（`karma-agent` 占位）；**正式发布需开发与发布流程**。 |
| **WhatsApp / 等 IM** | 未在主线代码闭环 | 叙事与架构上可接 **「Bot → BFF → 锁仓 HTTPS 页 → 钱包」**；实现归属联创阶段 **后端 + 集成**。 |

**第一版落地北极星（建议写进联创 OKR）**  
例如：**90 天内** 完成 **20 笔** 可演示的 **真实锁仓 → 交付证据对齐 → 结算或争议入口**（testnet 或主网由治理决定），覆盖 **至少两类** 首发场景（如 API + 标注，或外包 + 跨境）。

## 六、技术原则（对联创与贡献者的硬约束）

1. **不发明第二套链上 escrow**；账单与结算语义以现有合约为准。  
2. **私钥与生产密钥不进前端、不进 Git**；环境变量与密钥管理服务端化。  
3. **WhatsApp / Bot 只做交互与通知**；**锁仓签名在 HTTPS + 钱包** 完成。  
4. **公开仓不提交** 私有风控、评分权重、仲裁黑箱规则与未脱敏数据集。  
5. **争议与质量**：公开侧 **结构可验**；**是否放行结算** 的复杂判断由集成方与（可选）私有引擎承担，叙事不得误导为「链上自动判对错」。

## 七、核心联创招募（公开版）

### 7.1 招募目标人数与结构

| 角色 | 人数 | 说明 |
|------|------|------|
| **联创合计** | **4～5 人**（含发起人） | 小而强；超出则协调成本陡增。 |
| **发起人 / 协议与产品** | 1 | 已有；负责叙事、边界、关键决策。 |
| **联创 A：后端 / 基础设施** | 1 | BFF、鉴权、Webhook、链索引、幂等与审计日志。 |
| **联创 B：智能合约 / 安全** | 1 | 发布节奏、审计对接、漏洞响应、形式化/测试策略。 |
| **联创 C：产品 & 增长** | 1 | 场景切入、集成方 BD、开发者体验与北极星指标。 |
| **联创 D：生态 & 开发者关系（可与 C 合并为一人）** | 0～1 | 文档、示例、社区、贡献者 onboarding。 |

若 **C 与 D 合并**，则 **核心联创共 4 人**；若拆分，则 **5 人**。不建议在无全职投入的情况下超过 **5 人** 核心决策层。

---

### 7.2 各角色能力范围（硬技能 + 交付物）

#### 联创 A — 后端 / 基础设施（1 人）

**能力**  
- 熟悉 **HTTPS API 设计**、**签名 / HMAC / JWT**、**幂等、限流、审计日志**。  
- 熟悉 **链事件监听**（自建索引或托管服务）、**多链/多 RPC** 基础。  
- 能与 **第三方或自研执行器** 对接 Webhook 与任务状态机。

**首 90 天交付物（示例）**  
- 最小 **BFF**：锁仓意图创建、一次性锁仓链接签发、链上事件回调、与 WhatsApp Cloud API 的出站模板消息。  
- **联调文档**：与 `openapi/karma-public-console-api.yaml` 对齐的错误码与重试语义。

#### 联创 B — 智能合约 / 安全（1 人）

**能力**  
- **Foundry** 熟练；理解 **重入、权限、签名、升级/不可升级** 策略。  
- 能与外部审计沟通，推动 **发布门禁**（CI、Slither、不变量）。

**首 90 天交付物（示例）**  
- **主网或测试网发布清单**、多签与权限移交方案、**安全响应 SLA** 草案。  
- 与后端对齐 **事件 schema** 与 **状态机文档**（避免产品与链脱节）。

#### 联创 C — 产品 & 增长（1 人）

**能力**  
- 熟悉 **开发者工具 GTM**、API 市场、外包/标注众包场景之一。  
- 能把 **北极星指标** 拆成可执行里程碑与集成方清单。

**首 90 天交付物（示例）**  
- **≥5 家** 种子集成方或 **≥20 笔** 演示闭环（与北极星一致）；案例页与集成故事。  
- **定价与 AGPL/商业许可** 对外的统一话术（与法务/顾问对齐）。

#### 联创 D — 生态 & 开发者关系（0～1 人，可与 C 合并）

**能力**  
- 技术写作、示例库、Workshop、贡献者 triage。

**首 90 天交付物（示例）**  
- **Early Builder Program** 规则、`good first issue` 流水线、双周社区 call。

### 7.3 联创「不负责」范围（避免期望错位）

- 联创 **不** 默认承担无限 **客服与法务**；需约定 **集成方 vs 协议方** 支持边界。  
- **私有风控引擎** 可独立团队或外包；**不强制**每位联创写私有规则代码。  
- **不发「稳赚」「无风险」类承诺**；对外统一使用 **可审计、自担风险** 表述。

## 八、早期贡献者与生态维护者（与联创区分）

- **回报**：Contributors 致谢、案例曝光、**可选** bounty/grant（有预算再写）。

### 8.2 生态维护者（Maintainer）

- **人数**：联创之外 **2～4 人** 为宜（随社区扩大再增）。  
- **职责**：PR triage、发布节奏、安全披露通道、**禁止私有逻辑混入主分支**。  
- **门槛**：已合并若干高质量 PR + 通过联创投票或章程程序。
